协作优势

相比单 Agent 或人工团队，Multi-Agent 协作模式在多个维度有显著优势

1
职责清晰，永不越界
每个 Agent 有固定职责范围，PM 不写代码、Dev 不做测试、Test 不做架构设计。通过链路规则强制约束，角色之间不会混乱。
2
全自动交接，无需人工转交
每个 Agent 完成后，第一条消息自动 @ 下游角色继续处理。Dev 修完 bug 主动 @Test 继续测试，不依赖人工发现。
3
上下文不丢失
通过共享 AGENT_CONTEXT.md 文件，每个 Agent 都能读到项目当前状态。任何人中途加入都能立即恢复完整链路。
4
自然扩展，无需重新设计
加角色只需提供 SOUL 模板文件，setup.sh 自动识别并渲染。2~5 个角色任意组合，链路自动适配。
5
一键部署，零手动配置
用户只需提供飞书 App ID 和 App Secret，其余 profiles、SOUL.md、源码修复、进程启动全部自动化。
6
Dev↔Test 闭环，修复不过夜
Test 发现问题 → @Dev → Dev 修复 → @Test 继续验证 → 循环直到通过。所有问题在链路内闭环，不漏走到外部。

核心机制

关键技术设计，决定了 Multi-Agent 协作的可靠性

🧠
记忆持久化：AGENT_CONTEXT.md
每个 Agent 对话独立，无法共享上下文。我们设计了项目级共享文件机制：

【项目目录】/AGENT_CONTEXT.md

包含：原始任务 → 当前状态 → 链路记录 → 各环节输出 → 最终报告。

规则：每个 Agent 收到消息后立即读取文件了解全局状态，完成环节后更新文件并推进链路。任何人中途加入，从文件恢复。
🏷️
身份隔离：profile + SOUL.md 优先级
4 个 bot 连同一个飞书应用会共享全局 SOUL.md，身份必然混乱。

修复：修改 prompt_builder.py，profile SOUL.md 优先于全局 SOUL.md。每个 profile 有独立目录、配置、SOUL.md，各自加载专属身份，互不干扰。
🔗
链路接力：强制 @mention 第一条消息
协作最大风险：Agent 修复后不通知下游，链路断掉。

强制规则：Dev 修复完成后，第一条消息必须 @Test 通知继续验证，不允许先说"好的收到"。同样规则适用于 PM→Plan、Plan→Dev 的所有链路交接点。
🎯
意图识别：at_patterns 意图短语
LLM 经常生成"通知 PM"而不发真正 @mention，导致下游收不到。

解决方案：在 feishu.py 注册意图短语模式（"通知XX"、"汇报XX"、"艾特XX"），自动识别并插入 <at> XML 标签，确保每条意图都能触达。

问题规避

我们在真实场景中踩过的坑，已固化成自动化规避机制

Agent 身份丢失
4 个 bot 连到同一个飞书应用时，都会加载全局 SOUL.md，导致角色混乱。
修复 prompt_builder.py，profile SOUL.md 优先于全局 SOUL.md，每个 Agent 加载自己专属身份
@mention 中文名不触发
飞书默认只识别英文 @mention，LLM 生成的中文名字不会触发 bot。
在 feishu.py at_patterns 加入所有角色中文名，匹配 "通知XX"、"汇报XX" 等意图短语
Dev 修复后不通知 Test
Test 发现问题发给 Dev，Dev 修复后直接回复群里，Test 没收到通知，链路断在最后一公里。
Dev SOUL.md 强制规则：修复完成后必须第一条消息 @Test，不允许先说确认语
幽灵进程占用 app_id
启动时没有 --profile 参数，gateway 注册到所有飞书应用，导致后续启动全部冲突。
setup.sh 强制 pkill 旧进程 + 清理 lock 文件，每个 profile 独立进程 + --profile 参数
Session 缓存导致配置不生效
修改 SOUL.md 后测试，发现还是旧身份。Session 缓存了旧的系统提示。
修改 SOUL.md 后清 sessions 目录 + 重启 bot，或开新 DM 测试
应用未发布 WS 连上但无响应
飞书应用没有发布版本，WebSocket 连接成功，但 @消息完全无响应，排查困难。
在部署指南中明确强调必须发布版本，作为前置要求
